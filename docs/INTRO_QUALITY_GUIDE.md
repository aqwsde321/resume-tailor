# 자기소개 품질 가이드

- 문서 버전: v0.2
- 마지막 업데이트: 2026-03-07
- 기준 범위: `generate-intro` 결과 품질 기준

## 1. 목적

`generate-intro` 결과(`oneLineIntro`, `shortIntro`, `longIntro`)의 일관성과 신뢰도를 높이기 위한 기준 문서입니다.

## 2. 품질 원칙

1. 사실 기반
- `resume.json`에 없는 경력, 기술, 성과를 생성하지 않습니다.

2. 직무 정합성
- `company.json.requirements`를 먼저 보고, `preferredSkills`는 직접 근거가 있을 때만 보조적으로 반영합니다.
- 요약문보다 `projects`, `achievements`, `experience`, `strengths`를 우선 근거로 사용합니다.

3. 요건 앵커링
- 최소 1개 이상의 필수 요건을 소개글 본문에 직접 녹입니다.
- 가능하면 문장 흐름을 `공고 요건 -> 내 경험/성과/강점 -> 입사 후 기여`로 잡습니다.
- 우대 조건은 직접 연결 가능한 근거가 있을 때만 넣습니다.

4. 간결성
- `oneLineIntro`: 25~45자 권장
- `shortIntro`: 120~220자, 2~4문장 권장
- `longIntro`: 450~700자, 5~8문장 권장

5. 톤
- 전문적이고 담백한 톤을 유지합니다.
- 과장과 추상 표현은 최소화합니다.

## 3. 구조 템플릿

`oneLineIntro`

- 직무 + 핵심 역량 + 연차(또는 강점)

`shortIntro`

1. 직무 또는 연차 요약
2. 필수 요건 1개와 연결되는 핵심 프로젝트 또는 성과
3. 요구사항과 경험을 연결하는 문장
4. 선택적으로 입사 후 기여 방향

`longIntro`

1. 직무, 연차, 핵심 강점 요약
2. 대표 프로젝트와 성과 설명
3. 필수 요건과 직접 맞닿는 협업 방식 또는 문제 해결 경험 연결
4. 우대 조건 또는 추가 요구사항과 맞닿는 근거 구체화
5. 선택적으로 입사 후 기여 방향

`fitReasons`

1. 필수 또는 우대 요건 중 실제로 맞닿는 항목만 고릅니다.
2. 각 항목을 이력서의 프로젝트, 성과, 경력 근거와 직접 연결합니다.
3. 일반론 대신 구체적 사실을 씁니다.

## 4. 금지 규칙

- 검증 불가능한 수치 성과를 임의로 추가하지 않습니다.
- 회사명과 직무명을 잘못 쓰지 않습니다.
- 동일 의미 반복 문장을 2회 이상 쓰지 않습니다.
- 지나친 수식어(예: "압도적", "완벽한")를 피합니다.
- 공고 요건과 무관한 경험을 억지로 끼워 넣지 않습니다.
- 우대 조건에 직접 근거가 없는데 있는 것처럼 서술하지 않습니다.

## 5. 빠른 품질 체크리스트

- [ ] 회사명과 직무명이 `company.json`과 일치하는가?
- [ ] 필수 요건이 소개글 본문에 1개 이상 직접 연결되는가?
- [ ] 핵심 기술이 `requirements`와 연결되는가?
- [ ] 우대 조건은 직접 근거가 있을 때만 들어갔는가?
- [ ] 경력 연차가 `resume.json.careerYears`와 맞는가?
- [ ] 과장 또는 허위 문장이 없는가?
- [ ] 요약문만 반복하지 않고 프로젝트, 성과, 경력 근거가 보이는가?
- [ ] 문장 길이와 읽기 흐름이 자연스러운가?

## 6. 개선 루프

1. 사용자 피드백 수집
- 어색한 표현, 과장 여부, 직무 핏 부족 사례를 수집합니다.

2. 실패 케이스 분류
- 사실 오류
- 직무 정합성 부족
- 문체 또는 길이 문제

3. SKILL 가이드와 프롬프트 수정
- 실패 유형별 제약 문장을 보강합니다.

4. 재검증
- 동일 입력으로 전후 결과를 비교합니다.
